# Example — a finished context index

A generic illustration of the shape this skill produces. **The names and systems here are placeholders** — a real index uses the organization's actual tools and links. Two levels are shown: a top index page and one sub-page it links to.

Every entry follows the same shape: **`[Name (System)](link) — one short sentence saying what you'll find there.`** The sentence is what lets the AI (and a person) decide whether to open the link, so make it concrete, not telegraphic.

---

## Top index page

The page opens with a standard **purpose header** — boilerplate that says *what the page is*. It is the same for every deployment, lives in the index template, and is **not generated by `context-create`** (the skill leaves it untouched). Keep it verbatim, e.g.:

> **Context router for «Org».** Navigation, not storage — every entry is a pointer to where information actually lives. The AI reads this through the context tool and opens a branch only when needed (depth is fine). *(Optional one line on how the structure was derived, e.g. "Structure from the discovery assessment: N functions + systems map.")*

Then the content sections follow:

## Rules this example follows

1. **Pointers only** — name, link, one sentence saying what's there. Never paste content.
2. **Every entry earns its sentence** — concrete "what you'll find", not a bare label or a list of names crammed on one line.
3. **Nest growing areas into sub-pages** so the top page stays short and scannable.
4. **"Navigates itself?"** — a target with its own overview gets one link; a pile of files gets a short curated breakdown plus one fallback link.
5. **`needs link`** marks a known system whose link is still missing.
6. **No owner / access / sensitivity fields** — the source system enforces access.

# Example — a finished context index

A generic illustration of what this skill produces and how to classify before producing it. **The names and systems here are placeholders** — a real index uses the organization's actual tools and links.

Two things are shown:
1. A **classification cheat-sheet** (Phase 0) — how to decide *what level* a source belongs at.
2. The **finished shape** — a thin central index, one sub-index it links to, and a WRONG-vs-RIGHT contrast of the failure mode.

Every entry follows the same shape: **`[Name (System)](link) — one short sentence saying what you'll find there.`** The sentence is what lets the AI (and a person) decide whether to open the link — make it concrete, not telegraphic.

---

## 1. Classify before you build (Phase 0 cheat-sheet)

Open the source and classify on two axes — **KIND** (what is it?) and **SHAPE** (does it navigate itself?) — then route. Worked cases this skill has actually seen:

| Source (real example) | KIND — tell-tale | SHAPE | Route | Result |
|---|---|---|---|---|
| **"DLApp" space** — one product, many pages, but the space overview is empty/weak | PROJECT/AREA — *everything inside is about one product* | weak/empty | **create-subindex** | Build a Datalook sub-index on its OWN page; central gets **ONE** line. |
| **"KTI" space** — one client project, ~60 nested pages, home already links to its deals/SOW/diary/sub-pages | PROJECT/AREA — *one client project* | self-navigating (most sub-areas reachable in 1–2 clicks) | **link-only** | Add ONE line to central → the existing home. **Do not mirror.** |
| **Discovery / assessment doc** — lists many departments + systems | COMPANY-INVENTORY — *children are whole departments* | any | **create-central** | Build the thin central skeleton; push each area to its own sub-index. |
| **A Drive folder dump** — files, no overview | PILE-OF-FILES | any | **break-down** | Curated sub-index (key docs + one fallback folder link); ONE line in central. |

> **The rule the table encodes:** a single project is **always** a sub-index + one central line — **never** the central index itself. `create-central` is reachable only from COMPANY-INVENTORY. Multi-product tie-break: children that are themselves whole products → INVENTORY (recurse); children that all serve one product → PROJECT/AREA. Partial self-navigation → treat as weak/empty.

## 2. WRONG vs RIGHT — the failure this skill prevents

**WRONG — one project exploded into the central index** (what must never happen):

```
### Engineering
  Components — ...
  Portal — ...
  CML — ...
  AFT — ...
  Data — ...
  Business — ...
  Support — ...
```
☝️ This is *one project's* internal sections poured onto the company map. It should be **one line + a sub-index.**

**RIGHT — same content, correct altitude:**

```
### Engineering
  [Datalook (Wiki)](→ sub-index) — the Datalook product; its components, portal, CML, data and support all live on its sub-index.
```
☝️ **ONE line** on central. All that detail sits one level down, on the Datalook sub-index (shown below).

## 3. The finished shape

### Top index page (THIN — a map of maps)

The page opens with a standard **purpose header** — boilerplate that says *what the page is*. It is the same for every deployment, lives in the index template, and is **not generated by `context-create`** (the skill leaves it untouched). Keep it verbatim, e.g.:

> **Context router for «Org».** Navigation, not storage — every entry is a pointer to where information actually lives. The AI reads this through the context tool and opens a branch only when needed (depth is fine). *(Optional one line on how the structure was derived, e.g. "Structure from the discovery assessment: N functions + systems map.")*

Then the content sections — **every project/area is exactly ONE line that links onward:**

#### Company — cross-cutting: projects, strategy, people
*Things that span the whole org — start here when a request isn't tied to one team.*
- [Project Atlas](→ sub-page below) — the current flagship delivery; its plan, discovery and scope all live on its sub-index. *(Note: ONE line. All project detail is one level down.)*
- Team & roles — who leads which function and who to ask about what. `needs link`
- HR system — employee records, leave and internal policies (system not chosen yet). `needs link`

#### Engineering — how the product is built and run
- [Datalook (Wiki)](→ Datalook sub-index) — the data-platform product; components, portal, CML, AFT, data and support all live on its sub-index. *(ONE line — this is the exact spot the old skill exploded.)*
- [American Trading (Wiki)](#) — client project; its home already links to deals, SOW and the project diary, so we **link, we do not mirror** (no sub-page in this repo). *(Self-navigating → link-only.)*
- Services & libraries (Git) — supporting microservices and shared packages. `needs link`

#### Data — analytics, pipelines and reporting
- [Warehouse (data platform)](#) — the central analytics datasets and their schemas.
- Dashboards (BI tool) — the reports and metrics the business looks at day to day. `needs link`

#### Business — go-to-market, customers and money
- CRM (Sales) — the sales pipeline, accounts and deal history. `needs link`
- Helpdesk (Support) — incoming customer tickets plus the public help center. `needs link`
- Accounting / invoicing system — invoices, billing and financial records. `needs link`

## Rules this example follows
1. **Classify before building (Phase 0)** — a single project is always a sub-index + ONE central line, never the central index itself.
2. **Pointers only** — name, link, one sentence saying what's there. Never paste content.
3. **Only ONE link crosses up a level** — if a project's internal sections appear on the central page, the level was mis-classified.
4. **Self-navigating targets are linked, never mirrored** (see American Trading); weak/empty ones get a sub-index built first (see Datalook).
5. **Nest growing areas into sub-indexes** so the central page stays short and scannable.
6. **`needs link`** marks a known system whose link is still missing.
7. **No owner / access / sensitivity fields** — the source system enforces access.
8. **The purpose header is template boilerplate** — not generated by this skill; left untouched.
